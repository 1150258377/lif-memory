LIF-Memory · Visual Architecture

事件驱动的个人记忆回放系统

LIF-Memory 不是把 Obsidian 笔记压缩成摘要，而是把分散的笔记证据转化为状态电位；当长期状态积累到阈值时，触发 spike，并把它转化为一次可执行的诊断或最小实验。

EvidenceVector V_fast / V_slow Spike packet Profile-aware diagnosis Feedback closure

核心问题

不是“最近写了什么”，而是“哪个长期状态正在积累到足以重新打开的阈值”。

LIF controls triggering.
Profile improves interpretation.
LLM improves language.
Feedback closes the loop.

1. Pipeline

Profile 层不参与电压更新。它只在 spike 已经触发之后，帮助 LLM 更准确地解释这个 spike 为什么重要。

Obsidian notes保留原始完整信息
Evidence fragments抽取可追溯片段
EvidenceVector行动性、阻塞、完成、具体性
LIF voltageV_fast / V_slow 累积与泄漏
Spike packet阈值触发后的事件包
Profile-aware review只读背景解释，不改控制量
Minimal experiment下一步行动与反馈闭环

2. Layer boundaries

最重要的架构边界：LIF 层负责“什么时候触发”，LLM 与 Profile 只负责“如何解释和表达”。

Signal layer

从笔记中抽取证据片段，计算 EvidenceVector 和输入电流。

notes → vector

LIF control layer

更新电位、判断阈值、冷却、优先级和 action policy。

voltage → spike

Profile layer

提供紧凑、公开安全、长期稳定的语义先验。

context → interpretation

Feedback layer

通过 done、downgraded、ignored、postponed 关闭或调节主题。

closure → memory

3. Current state neurons

当前公开原型保留五个主状态。后续可以扩展，但只有当新状态有稳定输入、清晰阈值和独立完成目标时才值得加入。

Experiment

实验链路、设备、参数、波形、数据闭环。

Thesis

论文主线、图表、证据块、答辩故事。

Career

简历、岗位、项目表达、求职动作。

AI_Memory

AI 记忆系统、Agent、语义审查、工具链。

Health

恢复、精力、过载、行动前的系统供电。

4. Spike diagnosis

Profile-aware spike 不应该只给一句提醒，而应该形成一次小型诊断。

trigger_event这次 spike 被什么直接触发？
historical_echo它像过去哪个反复模式？
deep_tension暴露了哪个尚未解决的张力？
current_hypothesis当前最可能的机制是什么？
minimal_experiment下一轮最小测试或行动是什么？
observe_metric如何判断这次实验有没有效果？

5. v0.8 roadmap

下一版的目标不是重构全部系统，而是让 LLM review 变成 profile-aware diagnosis。

1

Add --profile-file

允许指定一个紧凑的 profile markdown 文件。

2

Load profile for review only

只传给 LLM review，不参与 LIF 电压和阈值计算。

3

Extend review schema

加入 trigger_event、historical_echo、deep_tension 等诊断字段。

4

Render diagnosis card

在 Markdown 报告里显示短诊断卡，最后落到最小实验。
