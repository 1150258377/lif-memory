# LIF-Memory v1.0 Architecture

## One-line definition

LIF-Memory is an event-driven personal memory dynamics system for Obsidian. It keeps original notes as the lossless memory substrate, maps note fragments into auditable evidence current, accumulates that current in fast/slow LIF states, and emits traceable action or insight spikes when a topic crosses threshold.

## Unified pipeline

```text
Obsidian 原始笔记
  -> evidence fragments
  -> EvidenceVector
  -> V_fast / V_slow LIF state
  -> threshold spike
  -> action card / insight card
  -> human closure feedback
  -> topic policy update
```

The project should be understood as one system, not as disconnected scripts. Earlier versions solved different parts of the same loop.

## Layer 1: Memory Source

**Purpose:** preserve the full information in Obsidian instead of compressing it into summaries.

Inputs:

- daily notes
- project notes
- folders
- tags
- wikilinks
- generated reports and feedback files

Principle:

```text
Original notes preserve content.
State voltage preserves trend.
Spike cards trigger attention.
```

This layer is implemented mainly by daily-note discovery in `lif_memory.py` and graph parsing in `obsidian_graph_miner.py`.

## Layer 2: Evidence Sensor

**Purpose:** convert raw text fragments into auditable input current.

The key object is `EvidenceVector`:

```text
target_weight
+ actionability
+ urgency
+ blocker
+ completion
+ specificity
+ novelty
+ confidence
```

This layer prevents the system from being a raw keyword counter. It also explains why the LLM reviewer should remain a semantic sensor rather than a controller.

The LLM may review whether a spike topic, primary state, secondary states, and completion target match evidence. It should not directly modify voltage, threshold, cooldown, priority, or action policy.

## Layer 3: LIF Dynamics

**Purpose:** model unresolved action pressure or explanatory tension as a leaky integrate-and-fire state.

Core state variables:

```text
V_fast: recent acute pressure
V_slow: long-running background pressure
V: weighted combination used for threshold crossing
```

Core operations:

```text
evidence input -> charge
elapsed time -> leak
completion signal -> inhibit
threshold crossing -> spike
post-spike reset -> reduce voltage
feedback/cooldown -> adapt future triggers
```

This is the actual identity of the project: note fragments become state current; unresolved pressure becomes voltage; spike cards become event output.

## Layer 4: Spike Interface

**Purpose:** emit the smallest useful output instead of another vague reminder.

A useful spike contains:

```text
topic
primary_state
secondary_states
priority
blocker_type
action_policy
completion_target
evidence_notes
voltage_model
feedback section
```

The spike should answer four questions:

1. What topic has crossed threshold?
2. Why did it trigger?
3. Should it continue, isolate, downgrade, or recover first?
4. What one small result would close it?

## Action profile and insight profile

LIF-Memory has two modes of meaning for voltage:

```text
action profile:  V = unresolved action pressure
insight profile: V = unresolved explanatory tension
```

The action profile produces cards such as experiment, thesis, career, AI memory, and health. The insight profile produces thought cards such as innovation claims, experimental closure, thesis closure, action bottlenecks, and economics questions.

## Boundary of v1.0

v1.0 should not keep adding modules before the system can consistently produce visible outputs.

A stable v1.0 should be able to produce:

- one daily top-spike card
- one full replay report
- one graph report
- one insight report
- one closure feedback update that persists into future runs
- one v1 convergence report generated by `lif_memory_v1_report.py`

## Relation to the BCI thesis

The software project mirrors the research project:

```text
BCI thesis:
continuous EEG -> LIF event -> backscatter transmission -> detectable rhythm

LIF-Memory:
continuous notes -> LIF memory event -> action/insight trigger -> evidence-based closure
```

The shared idea is event-driven compression without destroying the original source. In the thesis, the source is EEG. In LIF-Memory, the source is Obsidian memory.
