# Amygdala-Striatum LIF v0.9

This update adds a brain-inspired action-selection layer on top of the existing MazeGraph / LLM-MazeGraph pipeline.

The old pipeline already answers:

```text
Raw notes -> MazeGraph / LLM roles -> graph spikes / tasks / tensions
```

The new layer asks one more question:

```text
Given many candidate nodes, which one deserves fine inspection, testing, memory writing, delegation, backtracking, or dropping?
```

## Core idea

This version introduces four concepts:

```text
Coarse vision  -> fast, cheap, low-resolution scanning
Fine vision    -> slow, expensive, high-resolution inspection
Amygdala-like LIF population -> salience scoring
Striatum-like LIF population -> action selection
```

It keeps the system event-driven:

```text
candidate nodes
-> amygdala-like salience LIF
-> striatum-like action LIF
-> selected next action
-> memory / report update
```

## New file

- `amygdala_striatum_lif.py`

The module is standalone and can consume JSON generated by:

- `llm_maze_explorer.py`
- `maze_graph_lif.py`
- `maze_graph_builder.py`
- generic JSON packets containing nodes/tasks/spikes/claims/tensions

## What the amygdala-like population does

It is not a biological simulation. It is a symbolic LIF population for salience.

It contains six LIF channels:

| channel | meaning |
| --- | --- |
| `novelty` | new concept, new path, new bridge, uncommon term |
| `conflict` | contradiction, blockage, weak claim, missing link |
| `risk` | deadline, failure risk, hallucination risk, non-reproducible path |
| `value` | thesis value, job value, code value, evidence value |
| `urgency` | time pressure or immediate action pressure |
| `fatigue` | repeated, generic, stale, low-value material |

The output is a `salience_score` and a suggested mode:

```text
high salience -> fine vision
low salience  -> coarse scan / skip / summarize
```

## What the striatum-like population does

It converts salience into action selection.

Built-in actions:

| action | meaning |
| --- | --- |
| `fine_inspect` | deeply inspect this node or claim |
| `coarse_scan` | keep scanning broadly |
| `backtrack` | return to parent path or earlier evidence |
| `summarize` | compress into a report/card |
| `test_code` | run or generate tests |
| `write_memory` | preserve this as a memory node |
| `delegate_agent` | hand off to another LLM role |
| `drop_path` | stop spending attention here |

This gives LIF-Memory an explicit controller:

```text
not only "what spikes?"
but also "what should the system do after the spike?"
```

## Run

After generating a graph packet:

```powershell
python llm_maze_explorer.py --vault "." --days 30 --focus "今天最该处理什么" --json-output "llm_maze_graph.json" --output "LLM-MazeGraph 今日Spike.md"
```

Run the new controller:

```powershell
python amygdala_striatum_lif.py --input "llm_maze_graph.json" --output "Amygdala-Striatum LIF Report.md" --json-output "amygdala_striatum_lif.json" --top-k 12
```

Preview only:

```powershell
python amygdala_striatum_lif.py --input "llm_maze_graph.json" --dry-run
```

## Why this matters

Earlier LIF-Memory versions could generate spikes, but the next action after a spike was still implicit.

This update separates the control problem:

```text
MazeGraph / LLM roles = candidate generation
Amygdala-like LIF     = salience and danger/value detection
Striatum-like LIF     = action selection
Coarse/Fine vision    = compute-budget allocation
```

That makes the project closer to an autonomous knowledge explorer rather than a passive summarizer.

## Current limitations

- The current version uses symbolic word signals, not embeddings.
- The amygdala/striatum names are functional metaphors, not anatomical claims.
- LIF dynamics are lightweight controller dynamics, not full SNN simulation.
- The next version should allow `lif_memory.py` to directly consume `amygdala_striatum_lif.json`.
